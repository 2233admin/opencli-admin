# PRD: Collection Pack - Templated Multi-Resource Install with Requirement Checking

## Problem Statement

Standing up a new collection scenario (for example, an A-share quant data feed needing a specific Data Source, a Cron Schedule, a Browser Binding to a logged-in site, and sometimes an AI Agent / Model Provider) means an operator visits five unrelated CRUD screens - Sources, Schedules, Browsers, Agents, Providers - with nothing that declares what the scenario needs up front, checks whether those needs are already satisfied, previews what will be created, or reports afterward whether the scenario is actually healthy. `backend/api/v1/sources.py`, `nodes.py`, `browsers.py`, `schedules.py`, and `agents.py` are flat CRUD routers with no cross-object requirement-matching logic anywhere in their routes. The one place a multi-resource bundle is created in a single call - `apply_opinion_monitor_preset` in `backend/api/v1/presets.py` (line 143) - is fully hardcoded to one specific scenario (creates a Data Source, a Cron Schedule, and a Feishu Notification Rule) and does not touch edge nodes, browser bindings, AI agents, or model providers; a required Browser Binding is left as a manual follow-up hint in its `next` response field rather than being checked or created. Binding resolution itself only happens best-effort at run time - `backend/pipeline/pipeline.py` (`run_pipeline`, lines 132-158) looks up a `BrowserBinding` by site while a collection task is already running and silently proceeds without one if it is missing, rather than surfacing the gap before the operator commits to a scenario. There is no install-time capability check, no diff preview, and no post-install health signal anywhere in the codebase.

This confirms the CodeX handoff's core hypothesis - the UI/API cannot naturally express "install a collection pack" today - but the one existing precedent (`presets.py`) also shows the repo already has a working, if narrow, composite-install pattern worth generalizing rather than replacing.

## Solution

1. **Collection Pack Template catalog.** A small set of code-defined templates (same spirit as `backend/plan_ir/presets.py`'s adapter-metadata-driven `list_presets_grouped()`, and structurally the generalization of `presets.py`'s hardcoded opinion-monitor bundle): each template declares a name, description, the capability requirements it needs satisfied (edge node online with a matching runtime, a Browser Binding for a given site, an AI Agent / Model Provider of a given kind), and the resource blueprint it will create (Data Source config(s), Cron Schedule config(s), optionally a Notification Rule).
2. **Requirement check.** Before install, the template's declared requirements are checked live against existing `EdgeNode`, `BrowserBinding`, `AIAgent`, and `ModelProvider` rows, returning a per-requirement satisfied/missing checklist - the capability-matching step that does not exist anywhere in the codebase today.
3. **Install preview (diff).** A dry preview of exactly which Data Source / Cron Schedule / Notification Rule rows will be created (and which requirements, if any, are still missing and must be resolved first - e.g. bind a site before installing).
4. **Install.** Creates the blueprint's resources through the existing service layer (`source_service.create_source`, `schedule_service.create_schedule` - the same calls `presets.py` already makes), and records a `CollectionPackInstall` row linking the pack to the resource ids it created, so the install is traceable and reportable later.
5. **Post-install health.** Re-queries the linked resources' live status (source health, schedule enabled/last-run, binding still present) on demand - the health-reporting step the CodeX handoff asked for and that nothing today provides.

Pack install extends the `presets.py` pattern into a generic, requirement-aware system; it does not replace it, and it does not depend on the separate, partially-built Collection Canvas / Plan IR graph-authoring system (see Implementation Decisions).

## User Stories

1. As an operator, I want to browse a catalog of Collection Pack templates with name and description, so that I can find a scenario close to what I need without hand-assembling five resources.
2. As an operator, I want to see a template's requirement checklist (edge node, browser binding, agent, provider) before installing, so that I know what's already satisfied and what I still need to set up.
3. As an operator, I want a missing requirement to link to the existing screen that resolves it (Browsers page for a binding, Providers page for a model provider), so that closing a gap doesn't require guessing where to go.
4. As an operator, I want to preview exactly which Data Source, Schedule, and Notification Rule rows a pack will create before committing, so that I'm not surprised by what "install" actually does.
5. As an operator, I want install blocked (or clearly flagged) when a required capability is missing, so that I don't end up with a Data Source that silently has no browser binding, mirroring the run-time gap in `pipeline.py`.
6. As an operator, I want a successful install to return the ids of every resource it created, so that I can find and manage them afterward on their normal CRUD screens.
7. As an operator, I want to check a pack install's health after the fact (source status, schedule enabled, binding still present), so that I can tell a scenario is actually working without manually cross-checking four screens.
8. As an operator, I want pack templates and their requirements defined in code (not hand-entered through a UI), so that adding a new scenario is a reviewable code change, consistent with how `plan_ir/presets.py` already lists presets.
9. As an operator, I want the opinion-monitor bundle in `presets.py` eventually expressible as a Collection Pack template, so that the one hardcoded scenario the repo already has stops being a special case.
10. As a future maintainer, I want "Collection Pack" kept distinct from the unrelated `backend/browser_act_packs/` vendored skill library (which already uses `Pack`/`PackInfo` for a single browser-automation skill bundle), so that the two concepts don't collide in code or in operator-facing language.
11. As a future maintainer, I want Collection Pack kept out of the Collection Canvas / Plan IR graph-authoring surface (`docs/plan-ir-PRD.md`, ADR-0008/0009), so that a resource-provisioning concept doesn't get bolted onto a still-partially-wired graph system (per `docs/workflow-node-capability-mapping.md`: only 7 of 23+ catalog nodes have real backend runners today).

## Implementation Decisions

- **Templates are code, not DB rows.** A new `backend/collection_packs/catalog.py` defines a small, in-repo list of `CollectionPackTemplate` objects (key, name, description, requirements, resource blueprint) - matching how `plan_ir/presets.py` already derives its preset list from adapter metadata rather than a database table. Pack templates are a reviewable, versioned artifact; pack installs are the only thing that needs persistence.
- **`backend/models/collection_pack.py`** adds one model, `CollectionPackInstall` (`TimestampMixin`, matching every other model in `backend/models/`): `pack_key`, `resource_refs` (JSON list of `{resource_type, resource_id}`), so a later health check can re-query each linked row's live state rather than duplicating status into this table. This is deliberately much thinner than CodeX's original sketch implied - no separate "pack" resource graph, because the actual resources (Source, Schedule, Notification Rule) already exist and already have their own status.
- **`backend/schemas/collection_pack.py`**: `CollectionPackTemplateRead`, `CollectionPackRequirementCheck` (per-requirement satisfied/missing plus a link to the resolving screen), `CollectionPackInstallPreview`, `CollectionPackInstallRequest`/`Read`, `CollectionPackHealthRead`.
- **`backend/services/collection_pack_service.py`** holds the logic, per the thin-endpoint convention already documented in `providers.py`'s module docstring: `list_templates()`, `check_requirements(template, db)` (queries `EdgeNode`/`BrowserBinding`/`AIAgent`/`ModelProvider`), `preview_install(template, db)`, `install(template, db)` (calls existing `source_service.create_source` / `schedule_service.create_schedule`, then writes the `CollectionPackInstall` row), `get_health(install_id, db)`.
- **`backend/api/v1/collection_packs.py`** - thin router, `GET /collection-packs`, `GET /collection-packs/{key}/requirements`, `POST /collection-packs/{key}/preview`, `POST /collection-packs/{key}/install`, `GET /collection-packs/installs/{install_id}/health`. Named `collection_packs.py`, not CodeX's originally proposed `packs.py`, and the Pydantic/model classes are prefixed `CollectionPack*`, not bare `Pack*` - `backend/browser_act_packs/catalog.py` already defines `PackInfo`/`PackCatalog` for an unrelated concept (a single vendored browser-automation skill bundle), and reusing bare "Pack" risks real name collisions and operator confusion between two different products.
- **Frontend paths corrected.** CodeX's handoff referenced `frontend/src/pages/*.tsx`; that directory does not exist (confirmed via `git ls-files`). The real structure is Next.js App Router at `frontend/app/(app)/*/page.tsx` (e.g. `sources/page.tsx`, `schedules/page.tsx`). New pages: `frontend/app/(app)/collection-packs/page.tsx` (template gallery, `PageContainer` plus `EmptyState`/`ErrorState`/`LoadingState` plus `StatusBadge`, same pattern as `sources/page.tsx`) and `frontend/app/(app)/collection-packs/[key]/page.tsx` (requirement checklist, preview, install, health), both built on `@tanstack/react-query` hooks added to the existing `@/lib/api/endpoints` + `@/lib/api/hooks` layer. Note there is also currently no dedicated Browsers page in the frontend at all - a requirement pointing at "bind a browser" has nowhere to deep-link yet; flagged as a dependency, not solved by this PRD.
- **Deliberately not built on the Collection Canvas / Plan IR system.** That system (ADR-0008/0009, `docs/plan-ir-PRD.md`, `intelligence.input.collection-need` Canvas node, `GET /api/v1/workflows/capabilities`) solves a different axis - composing a data-flow graph - and per `docs/workflow-node-capability-mapping.md` most of its catalog nodes are still `BLOCKED`/`PREVIEW`, not wired to real backend runners. Collection Pack is a resource-provisioning concept (does this scenario's infra exist and work) with no graph-authoring shape; layering it onto a partially-built graph system would be a speculative abstraction the repo doesn't yet support. A pack MAY, once installed, be a reasonable seed for a future Plan, but that link is out of scope here.
- Everything lands on this branch's PRD only - no code changes ship in this commit.

## Testing Decisions

- **Primary seam - HTTP API integration** (`tests/integration/`, same layout as `test_providers_api.py`, `test_schedules_api.py`): new `test_collection_packs_api.py` covering template listing, requirement check (missing vs. satisfied), preview diff shape, install (asserts the created Source/Schedule ids and the `CollectionPackInstall` row), and health read.
- **Service unit seam** (`tests/unit/`, same layout as `test_schedules_service.py`, `test_source_service_filters.py`): `collection_pack_service` requirement-matching logic against fake `EdgeNode`/`BrowserBinding`/`AIAgent`/`ModelProvider` rows, and install's delegation to `source_service`/`schedule_service` (assert it calls the existing creators rather than duplicating their logic).
- Reuse the existing `presets.py` opinion-monitor bundle as a worked example once a template exists for it - a regression test that installing that template produces the same resources the hardcoded endpoint produces today.
- Acceptance bar matches repo convention: full pytest suite green, coverage gate holds, single alembic head (the `CollectionPackInstall` table needs one migration).

## Out of Scope

- Building the Browsers page in the frontend (does not exist today; a real gap this PRD surfaces but does not fix).
- Any change to the Collection Canvas / Plan IR system, or wiring a pack to auto-generate a Plan.
- Migrating the existing `apply_opinion_monitor_preset` endpoint to the new template system - noted as a follow-up, not required for the first pack template to ship.
- Uninstall / rollback of a pack install (delete the created resources by hand via their existing CRUD screens for now).
- Editing pack templates through the UI - they are code-defined, reviewed like any other change.
- Any new node-capability-declaration system on `EdgeNode`/`BrowserInstance` models - both are currently capability-free (confirmed by reading `backend/models/edge_node.py` and `backend/models/browser.py`); a template's "requirement" is matched against what the operator already configured (a site string, a runtime string), not a formal capability schema on those models. Building that schema is a bigger change than this PRD's first template needs.

## Further Notes

- Vocabulary per CONTEXT.md: this PRD introduces one new term, **Collection Pack**, deliberately positioned beside - not on top of - **Preset** (a single packaged node config for the Canvas palette, per CONTEXT.md's existing definition) and **Plan** (a graph). A Collection Pack is a scenario-level resource bundle; it may reference a Preset and may later seed a Plan, but is neither.
- Naming collision to keep in mind during implementation: `backend/browser_act_packs/` already uses "Pack" for an unrelated vendored concept (a single browser-automation skill). Keep all new identifiers prefixed `CollectionPack*` / paths under `collection_packs`, never bare `pack`/`Pack`.
- `docs/opencli-admin-backend-system-map.md` should be checked by the implementing agent before finalizing the service-layer wiring, per that doc's own stated purpose as the durable backend reuse map.
- The opinion-monitor precedent in `presets.py` is real, working code - the implementing agent should read it in full before writing `collection_pack_service.py`, since it is the closest existing analog for the install step's transaction shape (create Source, create Schedule, handle partial failure).
