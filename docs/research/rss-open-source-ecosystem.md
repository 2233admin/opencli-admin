# Open-Source RSS Ecosystem For OpenCLI Admin

Date checked: 2026-07-18.

Scope: open-source RSS source generators, feed catalogs, and automation frameworks that could expand OpenCLI Admin's `intelligence.source.rss` and workflow source catalog.

## Local Fit

OpenCLI Admin already has a first-class RSS/Atom workflow source: `backend/workflow/capability_projection.py` registers `intelligence.source.rss` as an RSS provider/channel and says Canvas Run executes RSS/Atom through `backend.workflow.rss_source_executor` while preserving `sourceGroup` lineage (`backend/workflow/capability_projection.py:153`, `backend/workflow/capability_projection.py:160`, `backend/workflow/capability_projection.py:164`, `backend/workflow/capability_projection.py:168`). The executor accepts `feedUrl`, enforces `allowed_domains`, and raises if the RSS host is not allowed (`backend/workflow/rss_source_executor.py:53`, `backend/workflow/rss_source_executor.py:69`, `backend/workflow/rss_source_executor.py:74`). The frontend contract requires `feedUrl` and `sourceGroup`, and explicitly states that the feed host must be allowed and output items retain `sourceGroup` lineage (`frontend/lib/workflow/node-contracts.ts:148`, `frontend/lib/workflow/node-contracts.ts:156`, `frontend/lib/workflow/node-contracts.ts:163`). The current financial RSS template already fans out multiple feeds into groups such as `macro-policy`, `market-regulation`, and `central-bank-research` before normalize, acceptance, and Records sink nodes (`frontend/lib/workflow/studio-templates.ts:172`, `frontend/lib/workflow/studio-templates.ts:176`, `frontend/lib/workflow/studio-templates.ts:181`, `frontend/lib/workflow/studio-templates.ts:185`, `frontend/lib/workflow/studio-templates.ts:190`, `frontend/lib/workflow/studio-templates.ts:194`, `frontend/lib/workflow/studio-templates.ts:217`, `frontend/lib/workflow/studio-templates.ts:219`).

Implication: prefer integrations that emit normal RSS/Atom URLs or OPML imports first. Treat browser-driven or authenticated generators as optional source-runtime services behind `allowed_domains`, not as a replacement for the local RSS executor.

## Recommendations

1. Start with `awesome-rss-feeds` as the low-risk feed catalog seed because it is static CC0 OPML data and maps cleanly to existing OPML/RSS source handling.
2. Add RSSHub as the primary generated-source catalog, but prefer self-hosted RSSHub routes for any workflow that matters operationally. Its route scale is the largest and its current maintenance signal is strongest, but AGPL-3.0 and route-specific auth/proxy needs require deliberate deployment boundaries.
3. Add RSS-Bridge for sites missing official feeds. It is lighter than RSSHub for single-site bridge generation and has permissive licensing, but WebDriver bridges and public-host rate limiting should be surfaced as runtime-resource requirements.
4. Use morss as an enrichment step for truncated feeds, not as the main catalog. It is useful where source feeds exist but item body quality is poor.
5. Use changedetection.io and Huginn as adjacent workflow integrations. They are valuable for browser steps, authenticated monitoring, webhooks, or multi-step event graphs, but they should not complicate the simple `intelligence.source.rss` node.
